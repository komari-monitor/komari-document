# 流量定时报告

流量定时报告会按日、周、月汇总节点流量，并通过已配置的通知渠道发送报告。它适合用于定期查看节点在上一天、上一周或上个月的实际流量使用情况。

## 前置条件

使用前需要先完成两项配置：

1. 在 **设置** → **通知** 中启用通知，并配置至少一个可用的通知渠道。
2. 在后台菜单进入 **通知** → **流量定时报告**，为需要报告的节点选择报告周期。

如果全局通知开关未启用，即使节点已配置日报、周报或月报，也不会发送流量报告。

## 报告周期

Komari 会自动注册三类定时任务：

| 类型 | 发送时间 | 统计范围 |
|------|----------|----------|
| 日报 | 每天 00:00 | 昨日 00:00:00 到 23:59:59 |
| 周报 | 每周一 00:00 | 上周一 00:00:00 到上周日 23:59:59 |
| 月报 | 每月 1 日 00:00 | 上一个自然月 |

时间范围按服务端本地时区计算。请确认运行 Komari 的服务器时区符合你的预期。

## 配置方法

在 **流量定时报告** 页面中，每个节点都有以下配置项：

| 字段 | 说明 |
|------|------|
| 启用 | 是否启用该节点的定时流量报告 |
| 日报 | 是否在日报中包含该节点 |
| 周报 | 是否在周报中包含该节点 |
| 月报 | 是否在月报中包含该节点 |

启用某个节点时，必须至少选择日报、周报、月报中的一种。否则系统会拒绝保存该配置，因为没有任何定时任务可以处理它。

## 流量计算方式

新版本的流量报告不再在发送报告时使用累计计数器的 `max - min` 反推流量，而是在每次写入节点状态记录时保存本次上报相对于上一条记录的精确增量。

这样可以避免两个常见误差：

- 节点重启或网卡计数器归零后，累计计数器从小值重新开始导致报表误算。
- 历史记录被压缩到长期表后，压缩记录不再是原始起点/终点快照，无法可靠反推周期流量。

长期历史记录压缩时会累加每个 15 分钟时间桶内的精确增量。压缩任务只会处理已经完整结束的 15 分钟桶，避免长期表中出现半个时间桶，导致报表读取时漏算同桶内仍保留的原始记录。

## 流量类型

报告中每个节点最终展示的流量值会参考该节点的 `traffic_limit_type` 设置：

| 类型 | 含义 |
|------|------|
| `up` | 只统计上行流量 |
| `down` | 只统计下行流量 |
| `sum` | 上行 + 下行 |
| `max` | 上行和下行中的较大值 |
| `min` | 上行和下行中的较小值 |

如果没有特殊需求，通常保持节点流量阈值类型的现有配置即可。

## 兼容性说明

精确增量依赖历史记录中的 `traffic_up` 和 `traffic_down` 字段。升级后新产生的记录会按新逻辑保存增量。

对于升级前已经存在的历史记录，这两个字段可能为空或为 0。它们不会自动回填，因此跨越升级时间点的早期报告可能只包含升级后产生的新增量。随着新记录持续写入，后续日报、周报、月报会逐步恢复准确。

## 常见问题

### 为什么启用时必须选择日报、周报或月报？

启用只表示该节点允许参与定时流量报告，日报、周报、月报决定它参与哪些定时任务。如果三者都不选，系统没有任何发送时机可以处理该配置，因此会拒绝保存这种状态。

### 为什么报告没有发送？

可以按以下顺序检查：

1. 全局通知开关是否启用。
2. 通知渠道是否可以成功发送测试消息。
3. 节点是否已启用流量定时报告。
4. 是否至少选择了当前报告类型，例如日报任务只会读取勾选了日报的节点。
5. 服务端时区是否符合预期，定时任务是否已经到达触发时间。

### 为什么刚升级后历史周期统计偏小？

旧记录没有精确增量字段，系统不会在本次修复中自动回填历史流量。升级后产生的新记录会正常参与统计，建议从升级后的完整周期开始观察报告结果。
